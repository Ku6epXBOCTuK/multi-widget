## Этап 1: Дизайн и протокол

- **Продумать визуальную структуру виджета**
  - Определить габариты контейнера под оверлей OBS (ориентир: 500x1000px).
  - Выбрать цветовую схему, адаптированную для работы на прозрачном или полупрозрачном фоне поверх стрима.

- **Спроектировать схемы данных**
  - Описать JSON-структуру сущности ToDo (идентификатор, текст, статус выполнения).
  - Спроектировать форматы WebSocket-событий для отправки изменений (добавление, обновление, удаление).

---

## Этап 2: Подготовка фронтенда

- **Настроить окружение фронтенд-проекта**
  - Развернуть чистый проект через менеджер пакетов вашего фронтенд-фреймворка.
  - Интегрировать выбранный CSS-инструмент (фреймворк или препроцессор).

- **Перевести сборщик в режим SPA (Single Page App)**
  - Подключить адаптер для генерации строго статических файлов (HTML/CSS/JS).
  - Отключить SSR (серверный рендеринг) в глобальной конфигурации проекта.
  - Настроить резервный роутинг на главную страницу (fallback для `index.html`).

---

## Этап 3: Статический прототип (Фронтенд в приоритете)

- **Создать компоненты интерфейса**
  - Написать изолированный компонент для строки задачи (текст, кастомный чекбокс).
  - Написать главный компонент-контейнер для рендеринга и вывода списка задач.
  - Добавить базовую CSS-анимацию для плавного появления и исчезновения строк при изменении статуса.

- **Настроить управление локальным состоянием**
  - Создать глобальное реактивное хранилище (стор) внутри фронтенд-приложения.
  - Наполнить стор статичным массивом mock-задач для немедленной проверки верстки.

- **Проверить отображение в стрим-софте**
  - Скомпилировать проект в финальную статическую папку (продуктовая сборка).
  - Добавить сгенерированный файл `index.html` в OBS Studio как источник «Браузер» и проверить читаемость.

---

## Этап 4: Разработка бэкенда

- **Развернуть архитектуру бэкенда**
  - Инициализировать систему рабочих пространств (воркспейс) для изоляции кода сервера и CLI-утилиты.
  - Создать проект сервера, подключить сетевой веб-фреймворк и асинхронный рантайм.

- **Реализовать постоянное хранение данных**
  - Написать модуль для чтения и записи состояния списка задач в локальный файл на диске или легковесную встроенную БД.
  - Обеспечить потокобезопасный доступ к данным в оперативной памяти сервера для конкурентных запросов.

- **Подготовить REST API слой**
  - Написать обработчик (Handler) для отдачи полного списка задач при первой загрузке (GET).
  - Написать обработчики для добавления новых задач и изменения статуса существующих (POST/PATCH/PUT).
  - Включить и настроить CORS-политику, чтобы локальный фронтенд из OBS мог без ограничений слать запросы на бэкенд.

---

## Этап 5: Интеграция WebSocket

- **Организовать WebSocket-сервер**
  - Добавить на сервере сетевой эндпоинт для апгрейда входящего HTTP-соединения до WebSocket.
  - Создать менеджер подключений для сохранения каналов связи со всеми активными окнами виджета.

- **Реализовать бродкаст и связать с фронтендом**
  - Интегрировать отправку сообщений в сокеты при каждом успешном мутировании данных через REST API.
  - Во фронтенд-приложении написать функцию начальной загрузки данных через fetch-запрос при старте.
  - Реализовать постоянное WebSocket-подключение на фронтенде с логикой автоматического переподключения (reconnect).

---

## Этап 6: CLI Утилита и сквозная проверка

- **Создать CLI приложение**
  - Добавить второй бинарник в воркспейс бэкенда и подключить библиотеку парсинга ввода.
  - Описать дерево команд и флагов интерфейса командной строки (например, `widget todo add`, `widget todo done`).

- **Настроить отправку команд и запустить тест**
  - Внедрить HTTP-клиент в CLI для отправки сформированных JSON-структур на локальный REST API сервера.
  - Провести финальный сквозной тест всей цепочки: ввод команды в консоли -> изменение в БД сервера -> бродкаст через сокет -> мгновенное обновление интерфейса в OBS.

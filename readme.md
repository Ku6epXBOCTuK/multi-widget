## Architecture Decision Record (ADR)

## Контекст

Необходимо создать расширяемую систему виджетов для OBS с управлением через терминал (CLI). Первым модулем будет интерактивный список задач (todo). Система должна быть универсальной: позволять на лету менять темы оформления, скрывать/показывать отдельные виджеты, менять глобальные настройки и легко масштабироваться под новые типы виджетов (например, chat, alerts) без переписывания архитектуры сервера.

## Архитектурные решения## 1. Единый CLI-интерфейс и структура команд

Вместо жестко прописанных утилит вводится иерархическая структура команд. Для реализации используется библиотека clap с вложенными подкомандами.
Синтаксис команд:

- `widget <имя_виджета> <действие> [параметры]` — управление конкретным модулем.
- `widget system <действие> [параметры]` — глобальное управление (темы, перезагрузка).

Примеры команд:

- `widget todo add "Сходить за хлебом"` - добавить задачу
- `widget todo done #2` - выполнить задачу
- `widget todo hide` - cкрыть виджет задач в OBS
- `widget system theme dark` - переключить глобальную тему на темную

## 2. Протокол обмена данными (Универсальный JSON Event-Driven)

Чтобы сервер и фронтенд не ломались при добавлении новых виджетов, коммуникация через WebSocket строится на базе Событий (Events), а не сырых массивов данных. Каждое сообщение — это JSON-объект с типом события.

```json
{
  "event": "todo:updated",
  "payload": [{ "id": 1, "text": "Задача", "completed": false }]
}
```

Для скрытия/показа и смены тем вводится системный тип события:

```json
{
  "event": "system:config",
  "payload": {
    "theme": "cyberpunk",
    "visibility": {
      "todo": false,
      "alerts": true
    }
  }
}
```

## 3. Модульная архитектура Axum (Бэкенд)

Сервер делит свое состояние на изолированные модули, объединенные в глобальный стейт через Arc.

- Каждому виджету выделяется свой префикс путей: `/api/v1/todo/_`, `/api/v1/system/_`.
- Единая точка рассылки (Broadcast Channel) передает любые структуры, реализующие Serialize.

Структура стейта в Rust:

```rust
struct GlobalState {
  system: Arc<Mutex<SystemConfig>>,
  todo: Arc<Mutex<TodoModule>>,
  // Сюда легко добавится новый виджет в будущем:
  // alerts: Arc<Mutex<AlertsModule>>,
  tx: tokio::sync::broadcast::Sender<WebsocketEvent>,
}
```

## 4. Универсальный фронтенд (SvelteKit)

Фронтенд собирается в один HTML-файл (vite-plugin-singlefile) для обеспечения отказоустойчивости (схема `file://`).
При этом роутинг реализуется динамически внутри одного экрана на основе состояния, полученного от сервера.

- По умолчанию виджет слушает WebSocket.
- При получении system:config корневой компонент Svelte меняет CSS-класс темы на <body> (например, class="theme-cyberpunk") и скрывает/показывает блоки в соответствии с объектом visibility.

## Последствия (Pros & Cons)

### Плюсы (Consequences Pros)

- Высокая масштабируемость: Чтобы добавить виджет «Оповещения», достаточно создать команду widget alerts send "Текст" и добавить обработчик JSON на фронтенде. Серверный транспорт WebSocket менять не нужно.
- Изоляция сбоев: Если CLI отправит некорректную команду для виджета todo, модуль system и другие виджеты продолжат работать.
- Кастомизация на лету: Смена темы через CLI мгновенно обновляет стили в OBS без перезапуска источника.
- Автономность: Если сервер выключен, HTML-виджет остается прозрачным и не портит стрим ошибками.

### Минусы (Consequences Cons)

- Усложнение парсинга на фронтенде: SvelteKit должен уметь обрабатывать разные типы событий в одном WebSocket-соединении (switch(event.type)), а не просто принимать один массив.
- Объем JSON-данных: При обновлении одной задачи сервер будет пересылать состояние модуля целиком (или диффы), что требует аккуратного проектирования структур в Rust.

## Как расширять этот комплекс в будущем

Если вы захотите добавить виджет «Счетчик смертей (Death Counter)»:

1.  В shared добавляете структуру DeathState.
2.  В cli добавляете ветку подкоманды: Commands::Death { sub: DeathCommands } (widget death inc).
3.  В server создаете эндпоинт `/api/v1/death/inc`, который обновляет счетчик и делает бродкаст:

```rust
tx.send(WebsocketEvent { event: "death:updated", payload: current_count });
```

4.  В Svelte-файле добавляете блок:

```svelte
{#if config.visibility.death}
  <DeathCounter count={deathCount} />
{/if}
```
